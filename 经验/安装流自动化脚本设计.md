# 安装流自动化脚本设计

## 脚本名称

`installations-automate`

该脚本仅适用于支持 WinGet 的 Windows 平台。

## 脚本参数

```powershell
installations-automate
installations-automate -s <file-path>
installations-automate --src <file-path>
installations-automate -l <file-path>
installations-automate --log <file-path>
installations-automate --reinstall [<file-path>]
installations-automate --check [<file-path>]
```

| 参数选项        | 选项说明 | 选项值           | 默认值                |
| --------------- | ---------------- | ---------------- | --------------------- |
| `-s` 或 `--src` | 指定安装流文件 | 安装流文件路径   | 无     |
| `-l` 或 `--log` | 指定输出日志文件 | 输出日志文件路径 | 无 |
| `--reinstall`   | 重新安装模式 | 日志文件路径 | installations-log.txt |
| `--check` | 检查模式 | 安装流文件路径 | installations.md |

> [!NOTE]
>
> - 如果未指定**安装流文件**，则默认为 `.\installations.md`；
> - 如果未指定**输出日志文件**，则默认为 `.\installations-log.txt`。

## 安装流文件

**安装流文件**应该是一个文本文件，语法类似 Markdown。

### 安装流名称

以 `# ` （一个井号一个空格）开头的行（即 Markdown 的一级标题）。

- 其后的文本（即 Markdown 的一级标题的内容）将作为**安装流的名称**，用于**输出消息**。

### 软件类别名

以 `## ` （两个井号一个空格）开头的行（即 Markdown 的二级标题）。

- 其后的文本（即 Markdown 的二级标题的内容）将作为**一个软件类别的名称**，用于**输出消息**。

### 软件名

以 `### ` （三个井号一个空格）开头的行（即 Markdown 的三级标题）。

- 其后的文本（即 Markdown 的三级标题的内容）将作为**一个软件的名称**，用于**输出消息**和**定位安装进度**。

### 安装信息

包含使用一对 `*` 包围的文本的行（即 Markdown 的斜体语法）。

- 其中使用一对 `*` 包围起来的文本（即 Markdown 的斜体语法的内容）将被识别为**安装信息**，并用于相关的**安装操作**；
- **安装信息**之前的文本将用于**输出消息**；
- **安装信息**之后的文本将被**忽略**。

> [!NOTE]
>
> 使用一对 `*` 包围的文本，不包括多个 `*` 的情况，比如 `**Q—Q**`（Markdown 的加粗语法）和 `***Q—Q***`（Markdown 的粗斜体语法）。所以如果你想要在安装流文件中使用斜体，请尽量使用**粗斜体**或 **HTML 语法**。

### 附加指令

使用

以三个 <code>`</code>（反引号）开头的行（即 <code>```...</code>，Markdown 的代码块语法前缀）

和

仅有三个 <code>`</code> 的行（即 <code>```</code>，Markdown 的代码块语法前缀）

围起来的多行。

> [!NOTE]
>
> 由于 Mrkdown 本身语法原因，此处不得不分成多行书写。

- 其中被围起来的**每行**文本，将**依次**被作为命令行命令**直接执行**。

### 普通文本

除以上几种特殊的行之外的所有行。

- 将被**原样输出**。

## 安装信息类型

安装信息有多种类型，是为了适应不同的安装操作，比如有些软件使用 WinGet 就能安装，一些不能。

安装信息不一定是和安装操作直接挂钩的，也可以是安装后的附加操作。

安装信息有**网页链接**，**WinGet ID** 和**文件路径**三种类型，对应于三种不同的操作。

| 安装信息类型 | 对应的操作                              |
| ------------ | --------------------------------------- |
| 网页链接     | 将使用系统默认浏览器打开此链接          |
| WinGet ID    | 将使用 `Winget install <ID>` 命令来安装 |
| 文件路径     | 将根据其类型打开或执行                  |

### 文件路径类型

文件路径可以指定一个文件夹，一个可执行文件，一个文本文件，一个 `.reg` 文件（通常用于恢复某个软件的设置），一个脚本文件等。针对不同的文件类型所执行的操作不同，如下表：

| 文件路径类型         | 对应操作                                                     |
| -------------------- | ------------------------------------------------------------ |
| 文件夹、文本文件     | 将使用 `explorer <path>` 命令在资源管理器中打开此文件夹或文件 |                  |
| 可执行文件、脚本文件 | 将使用 `<path>` 命令直接执行                                 |
| `.reg` 文件          | 将使用 `reg import <path>` 导入                              |
| 其他文件             | 将使用 `explorer /select, <path>` 命令在资源管理器中定位至此文件 |

> [!TIP]
>
> 建议路径使用相对于脚本文件的相对路径。

## 日志文件

日志文件主要用于**重新安装模式**使用，日志文件可以由**正常安装模式**、**重新安装模式**和**检查模式**产生，但逻辑应该都是一样的，以便**重新安装模式**，能够运作于不同的流程类型之后。

日志文件应该包含每个未安装（检查模式）或未能成功安装（正常和重新安装模式）的软件的全部信息。

日志文件应该以更简洁的（因为只有脚本使用）语言来记录信息。

日志文件应该记录以以下两种状态区分每个软件：

- 未知（通常是由于用户跳过了该软件的安装检查流程），重新安装模式针对此状态，应该先检查是否已安装，如果未安装才安装；
- 未安装（三种模式中经检查已确定的未安装的软件），重新安装模式针对此状态，直接重新安装；

由于安装流执行时间很长，为了避免意外状况导致的日志文件记录缺失，脚本应该实时将日志写入日志文件。

## 安装流程

每一个软件的安装流程，应该到下一个软件开始前才结束；

每一个软件的安装流程应该，包括**执行安装流文件指定操作**和**检查**两个部分；

### 执行安装流文件指定操作

对于第一个部分，如果涉及到打开链接、打开文件等需要用户手动干预的操作，脚本应该等待或允许用户跳过等待以开始下一个软件的安装。

如果用户跳过了等待，即意味着检查部分也被跳过，此时应该以**未知**状态记录到输出日志文件中，同时将检查操作所需的参数，记录到一个临时的文本文件中，以在脚本的最后针对这些软件再进行一次检查。

### 检查

对于检查部分，如果未成功安装，应该询问用户是跳过还是重试此软件的安装。

如果检查未成功，则记录到输出日志文件中，如果重试安装成功，则从输出日志文件中删除此软件的信息。

## 脚本执行流程

脚本最开始应该先设置编码为 UTF-8，以便能正确输出消息；

然后脚本处理参数；

然后进入安装流；

最终检查，该部分检查此前跳过检查部分的软件是否已安装，如果某个软件已安装，则需要从输出日志文件中，删除该软件的信息，如果未安装，则修改输出日志文件中的状态信息，然后询问用户是否重新安装，如果选择重新安装且成功，则从输出日志文件中删除该软件的信息。
