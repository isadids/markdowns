## ➤ 什么是图床？

### ➥ 图床的诞生

**Web**（万维网）以及 **HTML** 的诞生，开启了一个在文本中显示图片以及其他超越文本的内容（即**超文本**）的时代，后来的 Word（富文本）， Makdown 等都是基于 HTML 技术。

但是图片不是直接和文本一起存储的，而是分开存储，通过特定的语法在文本中**引用**。这也是计算机世界中不同类型的资源共同协作的标准解决方案，你不可能将所有资源都打包在一起，这样存储和传输效率都很低。

所以这种**在文本中引用图片**的方式，都有一个非常常见的问题，就是**一旦图片文件丢失，原有的对图片的引用就会失效**。尤其是对使用 Markdown 写作的人来说，我们希望将自己的文章通过互联网分享给他人，如果文章中引用了图片，那么我们就需要考虑将图片和文章一起分享，这样效率也很低，对方要访问我们的文章，就要将文章以及全部的图片一起下载。

当然 HTML 技术能够一定程度解决这个问题，因为 Web 天生就是自动加载引用的资源，用户只需要会上网就行，其他问题浏览器会解决。但是不是所有需要用到超文本技术的地方，都得用到 Web 来解决。简而言之，就是不可能让所有需要用到超文本技术的人，都是一个前端攻城狮。

而**图床**就是为了解决这个问题而诞生的，它利用云存储和 Web 技术，为图片创建一个**持久的** **URL**，你只需要在需要用到这张图片的地方，引用这个 URL 即可，因为只需要这个 URL，这个世界上的任何计算机就可以通过 Web 技术（HTTP 协议）访问到这张图片，就像在互联网中为这张图片分配了一块专用的存储空间和访问地址，而你**几乎**不必再考虑图片丢失问题。

### ➥ 图床的搭建

图床的搭建，无非就是解决以下问题：

- 在哪里集中存储图片？
- 如何才能在通过 URL 访问这些图片？

1. 在哪里集中存储图片？

- 首先存储图片的设备，需要在公网上是可访问的，其必须具备公网 IP，或者借助具有公网 IP 的设备来访问（通过内网穿透等方式）；
- 其次，该设备需要具备一定的稳定性，且全天 24 小时运行。

以上条件，显而易见，普通的家庭设备几乎是不行的，所以我们必须借助云存储服务。毫无疑问，一般的云存储服务都是收费的，不在本文的讨论范畴。

GitHub 为什么可以？

因为 GitHub 本身就是一种云存储服务，每个用户的每个仓库中的每个文件，都有其唯一的 URL。

2. 如何才能在通过 URL 访问这些图片？

将图片存储在公网可访问的设备上还不行，使用具体 IP 的 URL 无疑是几不稳定的，因此还需要一个域名，域名一般都是需要购买的，但是一些云存储服务自带域名，GitHub 本身就有自己的域名。
